It is a bit absurd when you put it that way. The file is unambiguously text — you could open it in Notepad — but the tool dispatches on the HTTP Content-Type header, and the server is sending application/javascript. The application/* prefix gets routed to the binary path regardless of what the bytes actually are. It's a classification-by-label rather than classification-by-content choice.

A small devil's-advocate nudge on the framing, though: "famously plain text" is doing some work there. By convention and by the bytes, yes. But MIME types were designed to tell clients how to handle content, not what encoding it uses — and application/javascript is actually the IANA-registered correct type for .js files (RFC 9239 deprecated text/javascript in favor of it, then partially walked that back). So the server is being pedantically correct, and the tool is being lazy about the application/* branch. The friction is real, but it's not quite the server's fault — it's that "is this human-readable?" and "what's the MIME type?" aren't the same question, even though we often act like they are.

The same thing happens with application/json, application/xml, application/x-yaml — all plain text, all routed as binary by naive handlers.

What it is

A single-page browser toy that simulates a defunct group-video-watching service (the <video> tag plus a chat panel sit side-by-side in the layout). One video — DehydrationLoop.mp4 — plays on loop, and a fake chat unfurls alongside it, with messages timed to the video's currentTime via a timeupdate listener. The whole simulation is essentially: as the video plays past timestamps 0, 10, 20, 30… seconds, a corresponding pre-scripted line from a fictional user (in the sample I read, parkerLotLost) gets appended to the chat. Each ChatItem is a subclass keyed to a screen name and avatar (iWantToBelieve, uMad, hatsuneMikuFan1997, asinineAssasin, etc.).

The clever bit is in index.js: it tracks latestSeen, and if the video's currentTime goes backward (the user scrubbed back or it looped), it calls reconcillePast, which increments timesLooped, wipes the chat, and re-renders. The author's design comments explicitly note this: "if you rewind, the chat rewinds as well. they notice. you've caught them in a loop." The infrastructure for swapping in different chat scripts on each loop is present (getCurrentStory(), the timesLooped % stories.length pattern in a commented-out line), though the live build only ships one story (north).

The HTML's meta description and the source comments say it directly: it's a Ludum Dare 51 entry (theme: "Every Ten Seconds"), and the keywords tag it as zampanio rabbit simulation ARG.
